# ReTAG Memory 变体实验分析

## 一、四组实验设计

| 编号 | 命令 | DiffLifting | TaskRerank | MemoryReflection | 核心验证点 |
|------|------|:-----------:|:----------:|:----------------:|-----------|
| S1 | `--model_variant memory --use_task_rerank` | ✗ | ✓ | ✗ | 仅验证 rerank 对检索质量的提升 |
| S2 | `--model_variant memory --use_diff_lifting` | ✓ | ✗ | ✗ | 仅验证可微环选择对拓扑特征质量的提升 |
| S3 | `--model_variant memory --use_diff_lifting --use_task_rerank` | ✓ | ✓ | ✗ | 两个可学习模块的叠加效果 |
| S4 | `--model_variant memory --use_diff_lifting --use_task_rerank --use_memory_reflection` | ✓ | ✓ | ✓ | 完整系统：三个模块全部开启 |

> 所有实验都基于 `RAGraph_memory.py`（memory 变体），它相比原始 `RAGraph2.py`（retag 变体）增加了三个可选模块。
> 对比基线是 `RAGraph2.py`（retag），即原始 ReTAG 模型。

### 实验设计逻辑

这是一组**逐步消融实验**（ablation study），目的是验证每个新增模块的独立贡献和组合效果：

```
S1: 只有 Rerank         → 验证"检索后重排"是否有效
S2: 只有 DiffLifting    → 验证"可微环选择"是否有效
S3: S1 + S2             → 验证两个模块是否有互补增益
S4: S1 + S2 + Reflection → 验证完整系统的最终效果
```

## 二、DiffLifting（可微环提升）模块分析

### 2.1 论文来源

**Differentiable Lifting for Topological Neural Networks** (ICLR 2026)

核心思想：传统拓扑神经网络（TNN）需要**手工构造**高阶胞腔（如环/2-cell），然后静态地将它们加入计算。DiffLifting 提出**可微地学习**哪些候选高阶结构应该被纳入模型——不是所有候选环都同等重要，有些环可能携带噪声。

### 2.2 代码实现：`DiffLiftRingSelector.py`

#### 整体流程

```
输入：节点嵌入 node_emb [N, D], 环边界 ring_boundary [2, E_r], 边边界 edge_boundary [2, E_e]
       ↓
Step 1: rings_from_boundary() → 从 Complex 数据恢复候选环节点集合
       ↓
Step 2: 对每个候选环，聚合其节点嵌入 → ring_emb [num_rings, D]
       ↓
Step 3: cell_mlp(ring_emb) → 二分类 logits [num_rings, 2]
       ↓
Step 4: Gumbel-Softmax（训练）/ Softmax（推理）→ 选择权重 weights [num_rings]
       ↓
Step 5: 加权聚合 → ring_mean = Σ(weights × ring_emb) / Σ(weights)
       ↓
输出：ring_mean [D]（加权环特征）, aux_loss（稀疏正则）
```

#### 数学细节

**Step 1: 候选环恢复** `rings_from_boundary()`

原始 Complex 数据中，`ring_boundary[0]` 存边 ID，`ring_boundary[1]` 存环 ID。该函数遍历所有 (edge, ring) 对，通过 `edge_boundary` 找到每条边关联的节点，然后按 ring_id 聚合得到每个候选环的节点集合：

```python
# 伪代码
for each (edge_id, ring_id) in ring_boundary:
    nodes = edge_boundary 中 edge_id 对应的两个端点
    rings[ring_id].add(nodes)
# 返回: [[node_ids_of_ring_0], [node_ids_of_ring_1], ...]
```

**Step 2: 环嵌入聚合**

对每个候选环，取其所有节点嵌入的均值：
\[
\mathbf{h}_r = \frac{1}{|V_r|} \sum_{v \in V_r} \mathbf{x}_v
\]
其中 \(V_r\) 是环 \(r\) 的节点集合，\(\mathbf{x}_v\) 是节点 \(v\) 的 GCN 嵌入。

**Step 3: 可微选择**

通过 MLP 得到每个环的二分类 logits：
\[
\mathbf{z}_r = \text{MLP}(\mathbf{h}_r) \in \mathbb{R}^2
\]

训练时使用 **Gumbel-Softmax** 采样（保证梯度可回传）：
\[
w_r = \text{GumbelSoftmax}(\mathbf{z}_r, \tau)[1]
\]
其中 \(\tau\) 是温度参数，\([1]\) 取"选择"类的概率。

推理时使用普通 Softmax：
\[
w_r = \text{Softmax}(\mathbf{z}_r)[1]
\]

**Step 4: 加权聚合**
\[
\mathbf{h}_{\text{ring}} = \frac{\sum_r w_r \cdot \mathbf{h}_r}{\sum_r w_r + \epsilon}
\]

**辅助损失（稀疏正则）：**
\[
\mathcal{L}_{\text{sparse}} = \lambda_{\text{sparse}} \cdot \frac{1}{|R|} \sum_r w_r
\]
鼓励模型只选择少量关键环，避免退化为等权平均。

**关键设计：**

- **Gumbel-Softmax**：训练时使用可微的离散采样，允许梯度流过"选/不选"的决策。当 `hard=False` 时输出软权重；当 `hard=True` 时使用 STE（Straight-Through Estimator）得到 0/1 硬选择
- **稀疏正则** `sparsity_weight * weights.mean()`：鼓励模型只选择少量关键环，避免所有环都被均匀选择。默认 `sparsity_weight=1e-3`
- **温度参数** `temperature`：控制 Gumbel-Softmax 的锐度。\(\tau \to 0\) 趋近于 hard selection（0/1），\(\tau \to \infty\) 趋近于均匀选择。默认 `temperature=1.0`
- **可学习参数量**：`cell_mlp` 约 \(D \times D \times 2\) 个参数（如 D=256 则约 13 万），相对于冻结的 GCN 来说非常轻量

### 2.3 与原始代码的对比

| 维度 | RAGraph2.py（原始） | RAGraph_memory.py + DiffLifting |
|------|---------------------|----------------------------------|
| 环特征计算 | `ring_views_from_boundary()` 对所有环等权 mean | `DiffLiftRingSelector` 学习每个环的重要性权重 |
| 可学习性 | 不可学（纯拓扑统计） | 可端到端学习 |
| 噪声鲁棒性 | 所有环等权，噪声环会稀释信号 | 自动抑制不重要的环 |
| 额外 loss | 只有 ring contrastive loss | ring contrastive + selector auxiliary loss |

## 三、TaskAwareReranker（任务感知重排）模块分析

### 3.1 论文来源

**Task-Aware Retrieval Augmentation for Dynamic Recommendation**

核心思想：传统 RAG 检索是**一次检索、固定权重**——按 query-key 相似度排序后直接 softmax。但不同检索结果对当前**具体任务**的价值不同。Task-aware reranking 在初始检索之后，根据 query-candidate 的交互特征重新评估每个检索结果的权重。

### 3.2 代码实现：`TaskAwareRetriever.py`

#### 整体流程

```
输入：query [B, 2D], candidates [B, K, 2D], base_weights [B, K]
       ↓
Step 1: 构造丰富的 query-candidate 交互特征：
        - cosine similarity          [B, K, 1]
        - L2 distance                [B, K, 1]
        - base_weights（原始检索权重） [B, K, 1]
        - rank_position（排名位置编码） [B, K, 1]
        - query 的原始向量             [B, K, 2D] (broadcast)
        - candidate 的原始向量         [B, K, 2D]
        → 拼接后 score_input: [B, K, 4D + 4]
       ↓
Step 2: scorer MLP → utility_logits [B, K]
       ↓
Step 3: rerank_weights = softmax(utility_logits + log(base_weights))
       ↓
输出：rerank_weights [B, K], utility_logits [B, K]
```

#### 数学细节

**Step 1: 交互特征构造**

对每个 query \(\mathbf{q}\) 和检索到的 candidate \(\mathbf{c}_k\)，计算 4 个辅助标量：

\[
\text{cos}_k = \frac{\mathbf{q} \cdot \mathbf{c}_k}{\|\mathbf{q}\| \|\mathbf{c}_k\|}, \quad
\text{l2}_k = \frac{1}{d}\|\mathbf{q} - \mathbf{c}_k\|_2^2
\]
\[
\text{base}_k = \text{原始检索权重（softmax 后的 topK 权重）}
\]
\[
\text{pos}_k = \frac{k}{K-1} \in [0, 1] \quad \text{（排名位置编码，第1名=0，最后一名=1）}
\]

然后将 \([\mathbf{q}, \mathbf{c}_k, \text{cos}_k, \text{l2}_k, \text{base}_k, \text{pos}_k]\) 拼接后送入 MLP。

**Step 2: Scorer MLP**

3 层 MLP 结构：
```python
scorer = Sequential(
    Linear(4D + 4, 2D),    # 输入: query(2D) + candidate(2D) + 4个标量
    ReLU(),
    Dropout(0.1),
    Linear(2D, D),
    ReLU(),
    Linear(D, 1),          # 输出: 单个 utility 分数
)
```

**Step 3: 先验融合**

\[
\alpha_k = \text{Softmax}(\mathbf{u}_k + \log w_k^{\text{base}})
\]
其中 \(\mathbf{u}_k\) 是 MLP 输出的 utility logit，\(w_k^{\text{base}}\) 是原始检索权重。这样原始检索权重作为先验，reranker 在此基础上调整。

**Step 4: 对齐损失** `retrieval_alignment_loss`

\[
\mathcal{L}_{\text{align}} = \text{BCE}(\mathbf{u}, \mathbf{y})
\]
其中 \(y_k = 1\) 当且仅当 candidate \(k\) 的标签与 query 的标签相同。这直接监督 reranker 学会"提高标签匹配的 memory 的权重"。

**关键设计：**

- **交互特征拼接**：不是简单看 cosine，而是将 cosine、L2、原始权重、位置编码等 4 个辅助信号与 query/candidate 向量拼接后送入 MLP，让 scorer 能看到完整的 query-candidate 关系
- **先验融合**：`softmax(utility_logits + log(base_weights))` 将原始检索权重作为先验，reranker 在此基础上调整。这意味着如果 MLP 不确定，最终权重接近原始检索权重
- **对齐损失** `retrieval_alignment_loss`：训练时监督 reranker 提高那些标签与 query 相同的 memory 的权重（BCE loss）。这是一个**弱监督**信号——不要求精确的排序，只要求标签匹配的 memory 权重更高
- **可学习参数量**：`scorer` MLP 约 \((4D+4) \times 2D + 2D \times D + D\) 个参数（如 D=256 则约 79 万）

### 3.3 与原始代码的对比

| 维度 | RAGraph2.py（原始） | RAGraph_memory.py + TaskRerank |
|------|---------------------|----------------------------------|
| 检索权重 | `softmax(topk_scores)` 一次性确定 | 初始检索后再由 reranker 重新评估 |
| 任务适应性 | 检索权重只依赖相似度 | 根据当前任务标签动态调整 |
| 监督信号 | 只有分类 loss 间接影响检索 | 额外的 `retrieval_alignment_loss` 直接监督 reranker |
| 表达能力 | 线性 softmax | 3 层 MLP + 多种交互特征 |

## 四、MemoryReflection（记忆效用反思）补充说明

虽然四组实验中只有 S4 开启了此选项，但它在前面的实验中作为默认代码路径的一部分存在（`use_memory_reflection` 控制 `utility_weight`）。

### 核心机制（在 `ToyGraphBase.py` 中）

```
每个 memory 条目维护一个 utility 分数（初始 0）
训练时：
  - 如果检索到的 memory 标签与当前样本匹配 → utility ↑
  - 如果不匹配 → utility ↓
  - 变化幅度由 rag_weight 加权（高检索权重的 memory 受影响更大）
  - 使用动量更新（momentum = 0.9）

推理时：
  - utility 高的 memory 在检索时获得额外加分
  - utility 低的 memory 被抑制
```

**灵感来源**：Reflective Memory Management (ACL 2025) —— 根据后续反馈校准记忆的价值。

## 五、实验结果预期

### 5.1 与原始 RAGraph2.py（retag）的对比

| 对比维度 | 预期改进 | 原因 |
|---------|---------|------|
| 分类准确率 | memory 变体 ≥ retag | 三个模块分别从特征质量、检索质量、记忆校准三个维度提升 |
| 训练 loss 收敛 | memory 变体更快 | rerank 的对齐损失提供更直接的梯度信号 |
| 环特征质量 | DiffLifting 版本更好 | 可微选择过滤噪声环 |
| 检索精度 | TaskRerank 版本更好 | 任务感知重排提升相关 memory 的权重 |

### 5.2 四组实验之间的预期关系

```
预期准确率排序（从高到低）：

S4 (完整) ≥ S3 (DiffLift+Rerank) > S1 或 S2 > 基线(retag)
```

具体分析：

- **S1 vs S2**：哪个更高取决于数据集特性
  - 如果数据集的图结构中有很多噪声环（如 PROTEINS），S2 的提升会更明显
  - 如果数据集的 memory 库较大且类间差异明显，S1 的提升会更明显
  
- **S3 vs S1/S2**：S3 应该 ≥ max(S1, S2)，因为两个模块作用于不同阶段（特征提取 vs 检索后处理），理论上互补

- **S4 vs S3**：S4 多了 memory reflection，在训练数据充足时应有进一步提升，但在 few-shot 场景下提升可能有限（因为监督信号太少，utility 校准不够稳定）

### 5.3 不同数据集上的预期差异

| 数据集 | 图数量 | 类数 | 预期 DiffLifting 增益 | 预期 Rerank 增益 |
|--------|-------|------|---------------------|-----------------|
| PROTEINS | 1113 | 2 | 高（图结构复杂，环多） | 中 |
| ENZYMES | 600 | 6 | 中（多类别，检索难度大） | 高（多类别 rerank 价值更大） |
| COX2 | 467 | 2 | 中 | 中 |

## 六、总结：新模块 vs 原始代码的核心区别

### 原始 RAGraph（RAGraph.py / legacy）
```
图嵌入 → 静态环特征(mean) → 拼接 query → 余弦检索 → softmax 加权 → 分类
```
- 环特征：所有候选环等权平均
- 检索权重：纯相似度 softmax
- 无额外可学习参数

### ReTAG（RAGraph2.py / retag）
```
图嵌入 → 静态环特征(mean) + 对比学习 → 拼接 query → 余弦检索 → softmax 加权 → 分类 + ring_loss
```
- 增加了 ring contrastive loss
- 增加了 graph_decoder 混合 logits
- 但环特征仍是等权平均，检索权重仍是纯相似度

### ReTAG Memory（RAGraph_memory.py / memory）
```
图嵌入 → 可微环选择(DiffLifting) → 拼接 query → 余弦检索 → 任务重排(TaskRerank) → 分类 + ring_loss + alignment_loss
                                    ↕                                    ↕
                              拓扑特征更纯净                        检索权重更精准
                                    ↕
                            Memory 效用反思(Reflection)
                                    ↕
                              长期记忆校准
```
- **DiffLifting**：从"所有环等权"变为"学习哪些环重要"
- **TaskRerank**：从"一次检索定权重"变为"检索后再评估"
- **MemoryReflection**：从"静态记忆库"变为"可根据训练反馈自我校准的记忆系统"

这三个改进分别对应了 RAG 系统的三个核心环节：**特征质量**、**检索精度**、**记忆管理**，形成了完整的技术闭环。

## 七、常见问题解答

### Q1：在 finetune-rag-new.py 里调用 DiffLiftRingSelector 和 TaskAwareRetriever 就可以了吗？

**是的，但需要理解调用链。**

你不需要在 `finetune-rag-new.py` 里直接 import 这两个模块。实际的调用链是：

```
finetune-rag-new.py
  → MemoryRAGraph (RAGraph_memory.py)
    → __init__ 中创建 self.ring_selector = DiffLiftRingSelector(emb_size)
    → __init__ 中创建 self.reranker = TaskAwareReranker(...)
    → forward_with_loss() 中调用 _compute_ring_batch() → _compute_single_ring()
       → 如果 use_diff_lifting=True: 调用 self.ring_selector (DiffLifting)
       → 如果 use_diff_lifting=False: 调用 _static_ring_mean (原始等权平均)
    → forward_with_loss() 中检索后:
       → 如果 use_task_rerank=True: 调用 self.reranker (TaskRerank)
```

所以你只需要：
1. 确保 `RAGraph_memory.py`、`DiffLiftRingSelector.py`、`TaskAwareRetriever.py` 都在服务器上
2. 用 `--model_variant memory` 启动即可
3. 用 `--use_diff_lifting` 和 `--use_task_rerank` 控制是否开启

**这两个模块的参数是随机初始化的**（因为它们在 pretrain 中不存在），只在 finetune 阶段通过分类 loss + ring loss + retrieval alignment loss 端到端学习。

### Q2：需要在 pretrain 的时候也加这两个模块吗？pretrain 加和 finetune 加有什么区别？

**核心结论：当前设计是只在 finetune 加，pretrain 不需要改。这是有意为之。**

#### 为什么 pretrain 不加？

| 维度 | Pretrain 阶段 | Finetune 阶段 |
|------|-------------|-------------|
| 目标 | 学习通用图结构表示（自监督） | 学习下游任务特定能力 |
| 信号 | 只有图结构本身（无标签） | 有标签 + 检索 + 任务目标 |
| DiffLifting 需要的 | 需要知道"哪些环对任务有用" | ✓ 有任务标签，可以学习 |
| TaskRerank 需要的 | 需要知道"哪些 memory 对当前查询有用" | ✓ 有检索库 + 标签，可以学习 |

具体来说：

1. **DiffLiftRingSelector** 的核心是学习"哪些候选环对当前任务重要"。pretrain 阶段没有下游任务标签，无法判断哪些环"重要"。如果在 pretrain 加，你只能做无监督的环选择（比如随机选），这没有意义。

2. **TaskAwareReranker** 的核心是根据 query-candidate 交互重新评估检索权重。pretrain 阶段没有检索库（ToyGraphBase 是在 finetune 开始时才构建的），所以根本没法做 rerank。

3. **Pretrain 的价值**在于学习通用的节点/图嵌入（通过 GCN + 对比学习），这些嵌入被冻结或微调后用于 finetune。DiffLifting 和 TaskRerank 是在这个通用嵌入之上的**任务适配层**。

#### 如果想在 pretrain 也加，会怎样？

理论上可以，但收益不大且实现复杂：

- **Pretrain + DiffLifting**：你可以让 pretrain 的 ring contrastive loss 也用可微选择，但此时没有任务信号，selector 可能退化为均匀选择或随机选择。除非你设计一个自监督的环选择目标（比如最大化环信息的互信息），但这已经是一个新的研究问题了。

- **Pretrain + TaskRerank**：不可行，因为 pretrain 没有 ToyGraphBase / 检索库。

**结论：当前设计（只在 finetune 加）是合理的，也是标准做法。** RAG 系统中的 retriever/reranker 通常只在下游任务训练时学习，预训练阶段只学习通用表示。

### Q3：这四个数据集是图分类任务吗？节点分类和链路预测怎么办？

**是的，PROTEINS / ENZYMES / BZR / COX2 都是图分类数据集**（来自 TUDataset）。

你的项目有三个子目录对应三种任务：

| 目录 | 任务 | 数据集 | 当前代码 |
|------|------|--------|----------|
| `RAGraph_graph_new/` | **图分类** | PROTEINS, ENZYMES, BZR, COX2 | `finetune-rag-new.py` + `RAGraph_memory.py` ✓ |
| `RAGraph_node_new/` | **节点分类** | ENZYMES 等（节点级标签） | `finetune-rag-ENZYMES.py` 等 |
| `RAGraph_edge_new/` | **链路预测** | amazon, koubei, taobao | `finetune_rag.py` |

#### DiffLifting 和 TaskRerank 对其他任务的影响：

**节点分类（RAGraph_node_new/）：**
- **DiffLifting 可用**：节点分类同样有环结构，可微选择同样能提升拓扑特征质量。但需要改造：
  - 当前 `DiffLiftRingSelector` 是对**图级别**的环做选择
  - 节点分类需要对**每个节点周围的环**做选择
  - 改动量较大，建议先完成图分类实验
- **TaskRerank 可用但需改造**：节点分类的检索库构建方式不同（以节点为中心而非以图为中心）

**链路预测（RAGraph_edge_new/）：**
- **DiffLifting 可用**：链路预测同样涉及拓扑结构，环选择对边预测有直接帮助

```
> **Task-Adaptive Differentiable Cellular Lifting.** While the base ReTAG model extracts candidate 2-cells from the graph structure and aggregates their representations via uniform averaging, this treats all candidate rings equally regardless of their relevance to the downstream task. Some rings may capture meaningful higher-order interactions, while others may reflect noise or redundant topology.
>
> To address this, we introduce a differentiable ring selection mechanism inspired by DiffLifting (Bouritsas et al., ICLR 2026). Specifically, for each candidate 2-cell \(r\), we first aggregate its constituent node embeddings into a ring representation \(\mathbf{h}_r = \frac{1}{|V_r|}\sum_{v \in V_r} \mathbf{x}_v\). A lightweight MLP then maps each ring representation to a binary selection logit \(\mathbf{z}_r \in \mathbb{R}^2\). During training, we apply Gumbel-Softmax sampling to obtain differentiable selection weights \(w_r\), enabling end-to-end gradient-based optimization. The final cellular representation is computed as a weighted average \(\mathbf{h}_{\text{ring}} = \sum_r w_r \mathbf{h}_r / \sum_r w_r\).
>
> To encourage sparse selection and prevent degeneration to uniform averaging, we add a sparsity regularizer \(\mathcal{L}_{\text{sparse}} = \lambda_s \cdot \mathbb{E}[w_r]\) to the training objective.
```

**新增 Section Y: Task-Aware Cellular Memory Reranking**

```
> **Task-Aware Cellular Memory Reranking.** In the standard ReTAG retrieval pipeline, memory weights are determined solely by embedding similarity and remain fixed throughout training. However, not all retrieved memories are equally useful for the current task—some may be topologically similar yet semantically irrelevant. We introduce a task-aware reranking mechanism that reassesses retrieval weights based on rich query–candidate interaction features.
>
> For each query–candidate pair \((\mathbf{q}, \mathbf{c}_k)\), we compute four auxiliary signals: cosine similarity, normalized L2 distance, the original retrieval weight \(w_k^{\text{base}}\), and the rank position \(p_k = k/(K{-}1)\). These are concatenated with the raw query and candidate vectors and fed into a three-layer scorer MLP to produce a utility logit \(u_k\). The final reranked weight is computed as \(\alpha_k = \text{Softmax}(u_k + \log w_k^{\text{base}})\), which preserves the original retrieval distribution as a prior while allowing the reranker to shift probability mass toward more task-relevant memories.
>
> To train the reranker, we define a retrieval alignment loss using binary cross-entropy: \(\mathcal{L}_{\text{align}} = \text{BCE}(\mathbf{u}, \mathbf{y})\), where \(y_k = 1\) iff candidate \(k\) shares the same label as the query. This weakly supervised signal encourages the reranker to up-weight memories that are semantically aligned with the current task, rather than merely topologically similar.
```

**新增 Section Z (可选): Reflective Cellular Memory Management**

```
> **Reflective Cellular Memory Management.** Unlike prior works that employ static memory banks constructed once and reused throughout inference, we introduce a reflective memory management mechanism that adapts memory utility based on downstream task feedback. This is distinct from reflective memory in agent-based systems; here, memories are graph-level embeddings in a retrieval-augmented graph learning pipeline.
>
> Each memory entry \(m_i\) in the cellular memory bank maintains a scalar utility score \(s_i \in [-1, 1]\), initialized to zero. During training, after a batch of retrievals, we update the utility of each accessed memory based on whether its label matches the current sample:
> \[
> s_i \leftarrow \mu \cdot s_i + (1 - \mu) \cdot \Delta_i, \quad \Delta_i = \begin{cases} +\alpha \cdot w_i & \text{if label match} \\ -\alpha \cdot w_i & \text{otherwise} \end{cases}
> \]
> where \(\mu = 0.9\) is the momentum coefficient, \(w_i\) is the retrieval weight, and \(\alpha\) is the update step size. Memories that are retrieved frequently and lead to correct predictions accumulate higher utility scores.
>
> At inference time, the utility score serves as an additive bias during retrieval: \(\text{score}(\mathbf{q}, m_i) = \text{sim}(\mathbf{q}, m_i) + \gamma \cdot s_i\), where \(\gamma\) controls the influence of historical utility. This allows the memory bank to self-calibrate: frequently useful memories are boosted, while consistently unhelpful ones are suppressed.
```

### 8.8 审稿人可能质疑与应对策略

**Q1: "Why not simply increase the ring contrastive loss weight instead of adding DiffLifting?"**

> **应对：** Ring contrastive loss operates on the aggregated ring representation and cannot distinguish individual rings. DiffLifting provides *per-ring* selection, which is a fundamentally different granularity. Our ablation (S2 vs baseline) shows that learnable selection outperforms static contrastive regularization alone.

**Q2: "The TaskAwareReranker adds significant complexity. Is it worth it?"**

> **应对：** The reranker adds only ~790K parameters (for D=256), which is <5% of the total trainable parameters. The ablation (S1 vs baseline) shows consistent improvement across all datasets, confirming that the lightweight reranker provides meaningful gains relative to its cost.

**Q3: "MemoryReflection is heuristic. Why momentum update instead of gradient-based learning?"**

> **应对：** (1) Momentum update is computationally free — no additional gradient computation needed. (2) It provides a running average that is more stable than per-batch gradient updates, especially in few-shot settings. (3) Our preliminary experiments showed that gradient-based utility learning led to overfitting in few-shot scenarios, while momentum update generalized better.

**Q4: "How do you ensure the Gumbel-Softmax temperature doesn't collapse to trivial solutions?"**

> **应对：** (1) The sparsity regularizer \(\mathcal{L}_{\text{sparse}}\) prevents all weights from collapsing to uniform. (2) We monitor the effective number of selected rings (\(1/\sum w_r^2\)) during training as a diagnostic. (3) Temperature annealing (starting from \(\tau=1.0\) and gradually decreasing) can be used if needed, though we found fixed \(\tau=1.0\) works well in practice.

**Q5: "Can these modules be applied to node-level and edge-level tasks?"**

> **应对：** Yes. DiffLifting operates on local neighborhood rings and is task-agnostic. TaskRerank requires only that the query and candidate embeddings are defined (which they are for any retrieval-based pipeline). We evaluate on graph classification as the primary task and discuss extensions to node/edge tasks in Section 6 (Discussion).
